# 长文本 TTS 优化实施方案

## 目标

把当前“整段串行生成 -> 播放 -> 再生成下一段”的链路，逐步升级为更适合长文本的方案：

1. **首包更快**：优先尽快产出并播放首段音频
2. **播放不停顿**：播放首段时，后台继续生成后续片段
3. **支持长任务**：整章/整本生成改为异步任务，不阻塞前台播放
4. **可缓存复用**：同书籍/章节/音色/倍速的结果可以命中缓存
5. **长文更连贯**：切片、上下文、跨段衔接都围绕“听感连续”优化

---

## 一、现状问题

当前主链路是：

1. 文本切片
2. 取当前片段
3. 同步请求 TTS 生成音频
4. 音频生成完成后开始播放
5. 播放结束后才开始下一段生成

这会导致：

- 长文本首段等待长
- 段与段之间容易出现空隙
- 整章/整本朗读无法后台持续推进
- 相同内容重复生成，成本高
- 缺少“长文任务”和“试听任务”的区分

---

## 二、目标架构

### Phase A：流式首包 + 分段队列播放（当前优先实现）

目标：即便暂时做不到真正的 HTTP 音频流式，也要先把结构改成：

- **首段优先生成**
- **首段一旦可播立即播放**
- **播放首段时后台预取后续 1~N 段音频**
- **下一段优先消费已预取结果**

执行链路：

```text
用户点击朗读
  ↓
切片生成 PlaybackQueueItem[]
  ↓
优先请求第 1 段音频
  ↓
第 1 段可播后立即开始播放
  ↓
播放期间后台预取第 2~N 段
  ↓
播放结束后优先读取缓存/预取结果
  ↓
继续下一段
```

### Phase B：异步整章/整本任务

把“试听/立即播放”和“后台长任务”分开：

- 前台：当前章节立即播放
- 后台：整章/整本生成任务进入任务表
- 后台任务可暂停/恢复/取消
- 任务结果可写入本地缓存或导出目录

### Phase C：缓存与连贯性

- 音频缓存索引（按 provider + voice + speed + 文本 hash）
- 段级缓存命中
- 章节级清单缓存
- 长文切片上下文增强（段首/段尾衔接）
- 连续段之间做静音裁剪、交叉淡入淡出或最小拼接

## 三、实施拆解

## 1. 流式首包

### 1.1 MVP 定义

当前阶段的“流式首包”定义为：

- 不强依赖服务端返回真正的流式音频字节流
- 通过**优先只生成首段**来缩短首播等待
- 在首段播放开始后，立即并发生成后续段

### 1.2 技术步骤

1. 在 `PlaybackService` 增加**音频预取缓存**
2. 增加“首段优先 / 后续后台预取”的调度逻辑
3. 新增播放状态字段，区分：
   - 首段预热
   - 后台预取
   - 已缓存段数
4. 后续若远端 provider 支持流式接口，再把“首段优先”升级为“真正边下边播”

### 1.3 后续真正流式的接口预留

后续可以把适配器统一成：

```ts
synthesize(request) => Promise<AudioAsset>
synthesizeStream(request) => AsyncIterable<AudioChunk>
```

MVP 阶段先保留 `Promise<audioPath>`，避免大改动。

---

## 2. 分段队列播放

### 2.1 MVP 要做的事

- 保留现有文本切片能力
- 增加“未来 N 段预取窗口”
- 下一段播放时优先读取预取结果
- 如果预取未完成，则回退为等待该段生成完成

### 2.2 推荐参数

- 切片长度：`220 ~ 320` 字（当前保留 280 字附近）
- 后台预取窗口：基础窗口 `2` 段，运行时按“平均生成耗时 / 平均播放耗时”动态扩缩，当前上限 `5` 段
- 首段优先：只要第 1 段可播就立刻进入播放
- 若生成明显慢于播放，会主动拉大窗口；若缓存积压充足，则缩小窗口避免无意义并发

### 2.3 失败回退

- 某段预取失败：只标记该段失败，不清空整个播放队列
- 真正播放当前段失败：停止队列并报错
- 用户停止朗读：取消当前版本号下的预取结果消费

---

## 3. 异步整章 / 整本任务

### 3.1 任务模型

建议增加 `tts_generation_tasks`：

- `task_id`
- `scope`（chapter/book）
- `book_id`
- `chapter_id`
- `provider_id`
- `voice_id`
- `speed`
- `status`（queued/running/paused/completed/failed/cancelled）
- `progress`
- `manifest_path`
- `error_message`

### 3.2 执行策略

- 前台试听和后台长任务使用同一套切片规则
- 后台任务写缓存，不占用当前播放进程
- 当前阅读章节优先级高于后台整本生成

### 3.3 交付顺序

1. 先支持“整章后台生成”
2. 再支持“整本后台生成”
3. 最后支持“恢复未完成任务”和“导出整本音频/清单”

---

## 4. 缓存策略

### 4.1 段级缓存键

建议缓存键：

```text
providerId + voiceId + speed + normalizedText + optionalStyle + modelVersion
```

再做 hash，映射到音频路径和元数据。

### 4.2 缓存层级

1. **内存缓存**：当前朗读会话的预取结果
2. **磁盘缓存**：跨会话复用的段级音频，当前已实现到 `~/.ai-novel-reader/tts-cache/`
3. **章节清单缓存**：记录章节由哪些段组成（待补）

当前实现说明：

- 缓存键由 `providerId + voiceId + speed + bookId + chapterId + title + normalizedText` 做 SHA-256 得到
- 音频文件落在 `audio/<hash-prefix>/<hash>.<ext>`
- 索引 manifest 落在 `index/<hash-prefix>/<hash>.json`
- 运行时 queue item 的随机 id 不参与键计算，因此跨会话、跨播放批次可稳定命中

### 4.3 失效规则

以下任一变化都应视为新缓存：

- provider / voice / speed 变化
- 文本归一化结果变化
- 模型版本变化
- 手动清缓存

---

## 5. 长文连贯性策略

### 5.1 切片策略

优先按以下顺序切：

1. 段落边界
2. 句号 / 问号 / 感叹号 / 分号
3. 超长句再按长度硬切

### 5.2 听感连贯

后续阶段建议补：

- 为每段生成保留少量前文摘要或尾句上下文
- 对段首和段尾静音做裁剪
- 控制相邻段的音量一致性
- 对章节切换插入轻提示音或短停顿，而不是长空白

### 5.3 章节级策略

- 章节标题不一定每段都重复读
- 章节切换时单独生成标题提示音频
- 角色声线类 provider 要尽量锁定同章节声纹参数

## 五、验收标准

### 文档验收

- 能清楚说明阶段目标、技术步骤和优先级
- 覆盖首包、队列、任务、缓存、连贯性

### 代码验收

- 播放逻辑不再是“每段播放完成后才开始下一段生成”
- 至少对可落盘音频 provider（远端 / 离线）支持后台预取
- 首段开始播放后，后台已经在生成后续片段
- 类型检查和测试通过
